# 0006. ティラノ作品間では開発基盤と接続原則を揃え、作品コードは共有しない

- 日付: 2026-07-21
- 状態: 採用

## 背景

arikoiに加え、negabokuとchiriyuku-monotachiもティラノスクリプトを実行基盤とする方針になった。3作品で同じ技術を採用するため、エンジン更新、セキュリティパッチ、開発サーバー、E2E、CI、配布手順を別々に考案・保守すると、同種の不具合や運用差が生じる。

一方、arikoiはブラウザ向け英語ビジュアルノベル、negabokuとchiriyuku-monotachiはJavaScript製戦闘処理を伴う日本語RPGであり、作品固有の要件は異なる。3作品を単一の共通ランタイムや汎用ゲームライブラリへ依存させると、一作品の変更が他作品のリリースや設計を拘束し、ADR 0005とAGENTS.mdの「作品リポジトリ」「汎用化しない」という方針にも反する。

## 選択肢

1. **作品間で何も揃えない**
   各作品は独立するが、ティラノ更新、既知のセキュリティ対応、E2Eの安定化など、同じ問題を作品ごとに解決する必要がある。
2. **3作品が依存する共通ランタイムまたは共通ゲームライブラリを作る**
   修正を一か所へ集約できるが、作品間の結合、バージョン調整、汎用化のための条件分岐と互換性維持が発生する。
3. **開発基盤と接続原則を揃え、実行物と作品コードは各作品が所有する（採用案）**
   共通の知見と操作体系を活用しながら、各作品を単独で開発・検証・配布できる。

## 採用案

- 各作品は、使用するティラノスクリプトのバージョンを固定し、エンジン一式、ライセンス、作品固有設定を自分のリポジトリで所有する。他作品の作業ツリーや未固定ブランチを実行時依存にしない
- ティラノの入手元、バージョン、エンジン更新手順、適用するセキュリティパッチと確認方法を同じ形式で記録する
- 開発コマンドは、該当する機能について `serve`、`e2e`、`release:*` など同じ名前と役割に揃える。Web版とデスクトップ版など配布先による実装差は許容する
- Playwright、CI、起動・ニューゲーム・セーブ/ロード等のE2E観点、文字送り完了後にセーブする待機方法は共通の技術と運用知識として再利用する。テストケースとセレクタは各作品が所有する
- `.ks`、UI、アセット、言語設定、ゲームデータ、ゲームルール、作品固有プラグインは各作品に置く
- RPG作品では「`.ks`の独自タグ → JavaScriptのシナリオ境界 → 作品固有処理 → ティラノ変数へ結果同期 → `.ks`へ復帰」という接続原則を揃える。戦闘状態はJSON直列化可能にし、シナリオからJavaScript内部オブジェクトを直接操作しない
- negabokuとchiriyuku-monotachiの戦闘実装は、仕様が似ていることだけを理由に共有しない
- chiriyuku-monotachiの第1作・第2作は同一シリーズ内のため、両作品で実際に使用し、同じ修正を反映したいことが確認できた処理に限り、同リポジトリの`shared/`で共有できる
- 共通ツールや再利用可能なCIへ切り出すのは、同じ変更が複数作品で繰り返され、独立したバージョン付きツールにする方が明確に保守コストを下げると確認できた後とする

## 採用理由

- 3作品でティラノ固有の知見、検証方法、セキュリティ対応を再利用できる
- コマンドと品質基準が揃うため、作品を移動しても同じ手順で起動・検証できる
- 各作品を単独で再現、更新、配布でき、他作品の変更や障害に巻き込まれない
- 共通ライブラリの公開API、互換性、リリース調整を先回りして設計せずに済む
- 作品固有の分岐を共通層へ蓄積せず、シナリオとゲームルールを各作品内で追いやすく保てる

## 欠点・リスク

- エンジン、E2Eヘルパー、CI設定などに意図的な重複が残る
- セキュリティパッチや更新手順を各作品へ反映し忘れる可能性がある
- 「同じ名前のコマンド」が作品ごとに少し異なる挙動を持つ可能性があるため、各READMEで配布対象と成果物を明記する必要がある
- 共通化の判断を遅らせるため、成熟後に同じ修正を複数リポジトリへ適用する期間が生じる

## 再評価条件

次のいずれかが成立した場合、この判断を見直す。

- 同一のエンジン更新またはセキュリティ修正を3作品へ繰り返し適用し、手作業による漏れや差異が実際に発生した
- E2Eまたは配布処理の同一コードが3作品で安定して使われ、作品固有の条件分岐なしに独立ツールへ切り出せる
- negabokuとchiriyuku-monotachiで、戦闘境界の公開APIとゲームルールが実装後も一致し、別実装を維持する方が明確に高コストになった
- ティラノスクリプトの公式配布・更新方法が変わり、各作品がエンジン一式を所有する方式が不適切になった

再評価しても、シナリオ、アセット、作品固有ルールを共通ランタイムへ移すことは別の判断とし、必要なら新しいADRを作成する。

## 影響範囲

- arikoi: `package.json`、Playwright E2E、CI、リリース処理、ティラノ更新手順。ADR 0005の「作品リポジトリ」「エンジン原則無改造」を維持する
- negaboku: ティラノ移行時のディレクトリ、JavaScriptプラグイン境界、E2E、CI、配布手順。negaboku側でも実装開始時に本判断と整合するADRまたは技術判断を記録する
- chiriyuku-monotachi: 各ゲームのティラノ構成と`shared/`の境界。シリーズ内共有は同リポジトリの既存基準を優先する
- 3作品共通: ティラノの版管理、セキュリティ対応、コマンドの役割、品質基準を可能な範囲で揃えるが、リポジトリ間の実行時依存は導入しない
